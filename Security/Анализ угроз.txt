Моделирование угроз состоит из шести последовательных шагов: 
    - определение активов;
        С точки зрения администратора баз данных идентификация ценных активов, которые должны быть защищены, заключается в идентификации конфиденциальной информации компании, 
        которая будет иметь коммерческий эффект, если она будет потеряна (недоступна) или украдена.
    - создание обзора архитектуры;
        Создание обзора архитектуры состоит из определения логической архитектуры приложения и выражения его в виде диаграммы (см. рис. Схема архитектуры.jpg) (указываеть серверы с их именем и IP-адресом), а также технологического стека.
                
        Это помогает выявить потенциально уязвимые области конечного приложения, а также выявить любые уязвимости, связанные с технологией.
        
        Пример, стека технологий:
            Технологии        Компонент                             Детали
            Active Directory  Аутентификация                        Аутентификация внутренних пользователей и административных команд.
                                                                    
            IIS               Веб сервер                            Аутентифицирует внешних пользователей и передает трафик на сервер приложений.
                                                                    
            .NET              Основные приложения и аутентификация  Основное веб-приложение, построенное с использованием платформы .NET.

                                                                    Обеспечивает аутентификацию форм для внутренних пользователей.

            SQL Server 2016   Уровень базы данных                   Базы данных, управляющие приложением, хранятся и управляются на экземпляре SQL Server 2016.

            IPsec             Криптография                          Данные шифруются в пути между сервером приложений и базы данных, используя IPsec.

            https             Протокол                              Внешние пользователи получают доступ к веб-приложению через https.
    - создание профиля безопасности;
        При создании профиля безопасности вы начинаете определять потоки данных, которые, в свою очередь, позволяют определять границы доверия и точки входа.
        
        Пример, профиля безопасности:
            Элемент профиля  Соображения
            Проверка ввода   Приложение запускает специальный T-SQL, в отличие от вызова хранимых процедур. Поэтому вход не может быть легко проверен на уровне SQL Server.
                             
                             Поскольку основной точкой входа является веб-сервер, границы доверия пересекаются, и вход не может быть доверен.
                             
            Аутентификация   Пользователи аутентифицируются в базе данных с помощью проверки второго уровня. Для доступа к базам данных не требуется аутентификация домена.
                             
                             Тестирование проникновения, чтобы убедиться, что учетная запись sa была отключена или переименована, не выполнялась на экземпляре.
                             
                             Сервер приложений разрешает учетные данные пользователя. Сервер приложений использует одного пользователя для аутентификации в базе данных.
                             
            Криптография     Данные шифруются при передаче через IPsec.
                             
                             Базы данных не шифруются с использованием TDE (прозрачное шифрование данных).
                             
                             Шифрование на уровне столбца не используется.
                             
            Аудит            SQL Audit не настроен, однако трассировка по умолчанию запущена, что приведет к захвату ограниченного поднабора активности, например создания новых объектов.
    - определение угроз;
        Определение потенциальных угроз в приложении. Обычно это предполагает проведение теста на проникновение.
    - документирование угроз (см. Методология STRIDE и DREAD.txt);
    - оценка и определение приоритетов угроз (см. Методология STRIDE и DREAD.txt).
