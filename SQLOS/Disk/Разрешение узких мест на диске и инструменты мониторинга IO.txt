Инструменты мониторинга.
    - Performance Monitor.
        PhysicalDisk
            % Disk Time
            Current Disk Queue Length
            Disk Transfers/Sec
            Disk Bytes/Sec
            Avg. Disk Sec/Read
            Avg. Disk Sec/Write
    - DMOs.
        sys.dm_io_virtual_file_stats
            Возвращает статистику ввода-вывода для файлов данных и журнала.
        
            https://docs.microsoft.com/ru-ru/sql/relational-databases/system-dynamic-management-views/sys-dm-io-virtual-file-stats-transact-sql
            
        sys.dm_os_wait_stats
            Возвращает данные обо всех случаях ожидания, обнаруженных выполнявшимися потоками (см. _Scripts\PAGEIOLATCH_%.sql).
            
            Ожидания, связанные с другими процессами ввода-вывода:
                ASYNC_IO_COMPLETION
                LOGBUFFER
                PAGEIOLATCH%
                WRITELOG
            
            https://docs.microsoft.com/ru-ru/sql/relational-databases/system-dynamic-management-views/sys-dm-os-wait-stats-transact-sql
        
Решения для устранения узких мест на диске.
    - Оптимизация нагрузки приложения.
        Оптимизация запросов, которые вызывают много дискового ввода-вывода.
    - Использование более быстрого пути ввода-вывода.
        Использовать диски, контроллеры и другую архитектуру с более быстрой передачей дисков в секунду.
        
        Нужно выяснить, что вызывает стресс на диске.
    - Использование RAID массива.
        Создание единого пула дисков для обслуживания всех файлов базы данных SQL Server, исключая файлы журналов транзакций (см. рис. _Picture\Конфигурации RAID.jpg).
        
        Эффективность пула накопителей зависит от конфигурации RAID-дисков.
    - Использование системы SAN.
    - Использование твердотельных дисков (SSD).
    - Правильное выравнивание дисков.    
    - Добавление системной памяти.
        Когда физической памяти мало, система начинает записывать содержимое памяти обратно на диск и чаще читает более мелкие блоки данных или читает большие блоки.
        Чем меньше памяти имеет система, тем больше используется дисковая подсистема.
    - Создание нескольких файлов и файловых групп.
        Разделяя таблицы, которые часто объединяются в отдельные файловые группы, а затем помещают файлы в файловые группы на отдельные диски, 
        разделенные пути ввода-вывода могут привести к повышению производительности.
        
        Для целей производительности и восстановления рекомендуется, чтобы при использовании нескольких файловых групп первичная файловая группа использовалась только для системных объектов, 
        а вторичные файловые группы должны использоваться только для пользовательских объектов.
    - Перемещение файлов журнала на отдельный физический диск.
        Файлы журнала транзакций SQL Server должны всегда, по возможности, располагаться на отдельном жестком диске из всех других файлов базы данных SQL Server. 
        Активность журнала транзакций в основном состоит из последовательных операций ввода-вывода, в отличие от непоследовательно (или случайного) ввода-вывода, необходимого для файлов данных.
        
        Разделение активности журнала транзакций от других неважных операций ввода-вывода диска может привести к повышению производительности ввода-вывода, 
        поскольку оно позволяет жестким дискам, содержащим файлы журналов, концентрироваться на последовательном вводе-выводе.
        
        Как правило, вы должны, по возможности, попытаться изолировать файлы с самым высоким уровнем ввода-вывода.
        Чтобы идентифицировать файлы с самым высоким уровнем ввода-вывода, обратитесь к DMO sys.dm_io_virtual_file_stats.
    - Использование секционированных таблиц.
