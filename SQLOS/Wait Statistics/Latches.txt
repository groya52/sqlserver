Эти ожидания отслеживаются и записываются SQL Server внутри DMV sys.dm_os_wait_stats и sys.dm_os_latch_stats.

Режимы защелок.
    SH (Shared)
        Используется, когда защелка считывает данные страницы.
    UP (Update)
        Используется всякий раз, когда страница нуждается в изменении, страница все еще может быть прочитана другими защелками.
    EX (Exclusive)
        Используется при изменении страницы. В отличие от защелки обновления, эксклюзивная защелка не разрешает чтение или запись другим режимам защелки.
    KP (Keep)
        Используется, для защиты страницы, чтобы она не могла быть уничтожена защелкой DT. Они совместимы с любым другим режимом защелки, за исключением режима DT.
    DT (Destroy)
        Используются при удалении содержимого из памяти, например, когда SQL Server хочет освободить страницу данных в памяти.
    
    Совместимость защелок:
        
            SH   UP   EX   KP   DT    
        SH  Yes  Yes  No   Yes  No
        UP  Yes  No   No   Yes  No
        EX  No   No   No   Yes  No
        KP  Yes  Yes  Yes  Yes  No
        DT  No   No   No   No   No
        
Буферные защелки. 
    Используются для защиты страниц данных в буферном кэше. 
    Они используются не только для пользовательских страниц данных, но также для системных страниц, таких как страница свободного пространства (PFS), которая отслеживает свободное пространство на страницах данных. 
    Внутри sys.dm_os_wait_stats они обозначаются PAGELATCH_[xx], где [xx] указывает режим защелки.

Небуферные защелки.
    Используются для защиты структур данных за пределами буферного кеша. 
    Внутри sys.dm_os_wait_stats они обозначаются LATCH_[xx], где [xx] указывает режим защелки.

IO защелки. 
    Используются, когда страницы данных считываются из подсистемы хранения в буферный кеш.
    Внутри sys.dm_os_wait_stats они обозначаются PAGEIOLATCH_[xx], где [xx] указывает режим защелки.
    
PAGELATCH_[xx]
    Ждет, когда защелка должна ждать, прежде чем он сможет получить доступ к странице в памяти.
    Основной причиной этих ожиданий являются другие защелки, которые уже находятся на странице и несовместимы с режимом защелки, который наш запрос хочет использовать.
    
    Частые причины появления:
        - PFS базы данных TempDB;
            Увелечение кол-ва файлов.
        - Утверждение вставки последней страницы.
            Секционирование, вычесление хеша и вставка строки в определенную секцию.
            
LATCH_[xx]
    sys.dm_os_waiting_tasks, в столбце resource_description можно указать дополнительную информацию о конкретном классе защелок.
    Если ожидания LATCH_[xx] не отображаются в sys.dm_os_waiting_tasks, но с высоким временем ожидания видны в sys.dm_os_wait_stats, то стартовая точка должна быть sys.dm_os_latch_waits.
    
PAGEIOLATCH_[хх]
    Пока SQL Server переносит страницу данных из подсистемы хранения в буферный кеш, на странице буфера будет помещена эксклюзивная защелка.
    Это гарантирует, что никакие другие параллельные транзакции не выделяют одну и ту же страницу буфера или одновременно пытаются перенести одну и ту же страницу данных из подсистемы хранения в буферный кеш.
    
    Режим защелки зависит от действия, которое инициировало перемещение страницы данных из подсистемы хранения в буферный кеш. 
    PAGEIOLATCH_SH будет записываться, если данные перемещаются доступ для чтения и PAGEIOLATCH_UP или PAGEIOLATCH_EX будет использоваться, если страница данных перемещается для изменения.
    
    Увелечение PAGEIOLATCH_[хх]:
        - Фрагментация индекса и устаревшая статистика;
        - SQL Server удалит страницы данных из кеша буфера, если они не были доступны в течение определенного таймфрейма, чтобы освободить место в кеше буфера;
            Добавление большего количества оперативной памяти в вашу систему увеличит количество страниц данных, которые может хранить буферный кеш.
            
            Cчетчики Perfmon, которые могут помочь получить представление об использовании кеша буфера:
                SQLServer:Buffer Manager\Buffer cache hit ratio
                SQLServer:Buffer Manager\Page life expectancy
        - Перезагрузка SQL Server (смотрим журнал SQL Server, чтобы  узнать, перезагружен ли SQL Server);
        - DBCC DROPCLEANBUFFERS (не записывается в журнал SQL Server);
        - Подсистема хранения.
            Perfmon
                PhysicalDisk\Avg. Disk sec/Read
                PhysicalDisk\Avg. Disk sec/Write
                PhysicalDisk\Disk Reads/sec
                PhysicalDisk\Disk Writes/sec
                PhysicalDisk\Disk Read Bytes/sec
                PhysicalDisk\Disk Write Bytes/sec
