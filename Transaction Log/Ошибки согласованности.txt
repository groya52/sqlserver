Если SQL Server обнаруживает страницу с плохой контрольной суммой или поврежденной страницей, 
она записывает страницы в базе данных msdb в таблице с именем dbo.suspect_pages.

    После записи подозрительной страницы в таблице dbo.suspect_pages SQL Server обновляет строку 
    после устранения проблемы путем восстановления страницы из резервной копии или с помощью DBCC CHECKDB. 
    
    Он также увеличивает количество ошибок каждый раз, 
    когда возникает ошибка с тем же самым событием event_type. 
    
    Вы должны контролировать эту таблицу для новых и обновленных записей, 
    и вы должны также периодически удалять строки из этой таблицы, 
    которые имеют event_type из 4 или 5, чтобы остановить заполнение таблицы.

К сожалению, параметры восстановления DBCC CHECKDB не поддерживаются в таблицах памяти. 
    Однако, когда вы берете резервную копию базы данных, содержащей файловую группу с оптимизированной памятью, 
    выполняется проверка контрольной суммы в отношении файлов в этой файловой группе. 

    Поэтому крайне важно, чтобы вы не только выполняли регулярные резервные копии, 
    но и проверяли, что их можно восстановить успешно, на регулярной основе. 
    
    Это связано с тем, что ваш единственный вариант в случае испорченной таблицы, 
    оптимизированной для памяти, заключается в восстановлении с последней известной хорошей резервной копии.
   


Чтобы восстановить системные базы данных, вам нужно запустить настройку. Когда вы восстанавливаете системные базы данных с помощью настройки, доступны параметры, описанные в таблице 8-3 .
   
605
    Может указывать на одну из двух проблем, в зависимости от серьезности ошибки.
    
    Если уровень серьезности ошибки 12, то это указывает на грязное чтение.
        
        Грязное чтение является транзакционной аномалией, 
        которая происходит, когда вы используете уровень изоляции READ UNCOMMITTED или подсказку запроса NOLOCK.
        
        Это происходит, когда транзакция считывает строку,
        которая никогда не существовала в базе данных,
        из-за отката другой транзакции.
    
        Чтобы устранить эту проблему, повторите запрос до тех пор, пока он не удастся или перепишите запрос, 
        чтобы избежать использования уровня изоляции READ UNCOMMITTED или подсказки запроса NOLOCK.
        
    Однако ошибка 605 может указывать на более серьезную проблему, 
    и часто она указывает на аппаратный сбой.
    
    Если уровень серьезности ошибки 21, страница может быть повреждена, 
    или некорректная страница может быть подана из операционной системы.
        
        Вы должны связаться с вашей командой хранения, 
        чтобы решить проблему с хранилищем. 
        
        Если это так, вам нужно либо восстановиться из резервной копии, 
        либо использовать DBCC CHECKDB, чтобы исправить проблему.
        
823
    Возникает, когда SQL Server пытается выполнить операцию ввода-вывода, и API Windows, 
    который он использует для выполнения этого действия, возвращает ошибку в Database Engine. 
    
    Ошибка 823 почти всегда связана с проблемой аппаратного обеспечения или драйверов.
    
    Если возникает ошибка 823, вы должны использовать DBCC CHECKDB,
    для проверки согласованности базы данных и любых других баз данных,
    которые находятся на том же томе. 
    
    Вы должны связаться с вашей командой хранения, чтобы решить проблему с хранилищем. 
    Администратор Windows также должен проверить журнал событий Windows для коррелированных сообщений об ошибках. 
    
    Наконец, вы должны либо восстановить базу данных из резервной копии, 
    либо использовать DBCC CHECKDB, чтобы исправить проблему.
        
824
    Если вызов в Windows API завершается успешно, 
    но с возвращенными данными возникают логические проблемы согласованности, возникает ошибка 824. 
    
    Как и ошибка 823, ошибка 824 обычно означает, что есть проблема с подсистемой хранения. 
    
    Если возникает ошибка 824, вы должны следовать тем же самым действиям, 
    что и при возникновении ошибки 823.
    
5180
    Происходит, когда обнаружен идентификатор файла, который недействителен.
    
    Идентификаторы файлов хранятся в указателях страниц,
    а также на системных страницах в начале каждого файла.
    
    Эта ошибка обычно вызвана коррумпированным указателем внутри страницы, 
    но также может указывать на проблему с компонентом Database Engine. 
    
    Если вы испытываете эту ошибку, 
    вы должны восстановить ее из резервной копии или выполнить DBCC CHECKDB, 
    чтобы исправить ошибку.
    
7105
    Возникает, когда строка в таблице ссылается на структуру большого объекта (Large Object Block), 
    которая не существует. 
    
    Это может произойти из-за грязного чтения так же, как ошибка серьезности 605, 
    или это может произойти в результате поврежденной страницы.
    
    Коррупция может быть либо на странице данных,
    которая указывает на структуру LOB,
    либо на страницу самой структуры LOB.

    Вы должны запустить DBCC CHECKDB, чтобы проверить наличие ошибок. 
    
    Если ошибки не были обнаружены, ошибка, скорее всего, 
    будет результатом грязного чтения.
    
    Однако, если вы обнаружите ошибки, либо восстановите базу данных из резервной копии, 
    либо используйте DBCC CHECKDB, чтобы исправить проблему.
