Опция Delayed Durability делает операцию фиксации асинхронной.

Клиент получает подтверждение о том, что транзакция совершена немедленно, 
не дожидаясь, чтобы запись фиксации была зафиксирована на диске.

Запись фиксации сохраняется в буфере журнала до тех пор, пока его содержимое не будет очищено, что происходит в одном из следующих случаев:
    - Блок журнала заполнен.
    - Фиксация долговременной транзакции в одной и той же базе данных.
      Запись фиксации из такой транзакции сбрасывает содержимое блока журнала на диск.
    - CHECKPOINT.
    - Выполняется успешно процедура sys.sp_flush_log.
    
Если SQL Server crashed до того, как запись фиксации была зафиксирована на диске, 
изменения данных из этой транзакции будут отменены при восстановлении, как если бы транзакция никогда не совершалась вообще.

Тем не менее, другие транзакции смогут увидеть изменения данных, 
выполненные такой транзакцией, между моментом фиксации и сбоем SQL Server.

Потеря данных также возможна при регулярном завершении работы SQL Server.
Несмотря на то, что SQL Server пытается сбросить буферы журналов во время выключения, нет гарантии, что эта операция будет успешной.

В некоторых редких случаях процесс CHECKPOINT может очищать грязные страницы данных до того, 
как записи журнала транзакций зафиксированы.
Если бы SQL Server crashed ровно в один и тот же момент, это привело бы к тому, 
что после перезагрузки база данных была повреждена и была бы несовместимой с транзакциями.

Параметр базы данных DELAYED_DURABILITY, контролирует поведение Delayed Durability в области базы данных:
    
    ALTER DATABASE { database_name | CURRENT } SET DELAYED_DURABILITY = { DISABLED | ALLOWED | FORCED }
        
        DISABLED 
            При использовании этой настройки все фиксируемые в базе данных транзакции являются полностью устойчивыми 
        FORCED
            При использовании этой настройки устойчивость каждой транзакции определяется на уровне транзакций.
        ALLOWED
            Если выбран этот параметр, все транзакции, которые фиксируются в базе данных, являются устойчиво отложенными.
            
В случае кросс-баз данных или распределенных транзакций все транзакции полностью прочны независимо от их настроек.

То же самое относится к технологиям отслеживания изменений и изменения данных. 
Любая транзакция, которая обновляет таблицы, которые включены для любой из этих технологий, будет полностью долговечной. 

Отсроченная долговечность также не поддерживается транзакционной репликацией.

Можно контролировать долговечность транзакций, указав режим продолжительности работы в операторе COMMIT:

    BEGIN TRAN;
    ...
    COMMIT WITH (DELAYED_DURABILITY = ON);
    
Группы доступности AlwaysOn будут получать записи фиксации только после того, 
как эти записи будут зафиксированы в журнале первичного узла и будут переданы по сети.
