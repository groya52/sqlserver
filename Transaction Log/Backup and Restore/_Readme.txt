ТИПЫ РЕЗЕРВНЫХ КОПИЙ:
    - ПОЛНОЕ РЕЗЕРВНОЕ КОПИРОВАНИЕ БАЗЫ ДАННЫХ.
      Эта полная резервная копия может быть использована для восстановления базы данных до состояния, 
      в котором она была, в момент завершения резервной копии базы данных.
      
      SQL Server позволяет обновлять базу данных при полной резервной копии. 
      Он отслеживает изменения, происходящие во время резервного копирования, 
      путем захвата части журнала транзакций в резервной копии базы данных. 
      
      Резервная копия также записывает номер последовательности журнала (LSN) при запуске резервного копирования базы данных, 
      а также LSN, когда завершается резервное копирование базы данных.	 
    - ДИФФЕРЕНЦИАЛЬНЫЕ РЕЗЕРВНЫЕ КОПИИ БАЗ ДАННЫХ.
	  Дифференциальные резервные копии базы данных фиксируют изменения в любых объемах данных, 
	  которые произошли со времени последней полной резервной копии базы данных.
	  
	  Когда выполняется резервное копирование дифференциальной базы данных, 
	  в резервную копию записываются только те экстенты, которые имеют модифицированные страницы.
    - Частичное резервное копирование.
	  Частичные резервные копии обеспечивают средство для удаления данных только для чтения из резервной копии.
	  
	  В некоторых реализациях часть данных в базе данных может не изменяться и строго используется для запроса. 
	  Если эти данные помещаются в файловую группу только для чтения, 
	  вы можете использовать частичное резервное копирование для резервного копирования всего, 
	  кроме данных только для чтения.
    - ДИФФЕРЕНЦИАЛЬНЫЕ ЧАСТИЧНЫЕ РЕЗЕРВНЫЕ КОПИИ.
	  Дифференциальные частичные резервные копии работают как дифференциальные резервные копии баз данных, 
	  но ориентированы на данные того же типа, что и на частичное резервное копирование.
    - РЕЗЕРВНЫЕ КОПИИ ФАЙЛОВ И ФАЙЛОВЫХ ГРУПП.
	  Резервные копии файлов и файловых групп предназначены для баз данных, 
	  содержащих более одной файловой группы.

	  Резервная копия файловой группы, все файлы, 
	  определенные в файловой группе, резервируются.
    - РЕЗЕРВНЫЕ КОПИИ ТОЛЬКО ДЛЯ КОПИРОВАНИЯ.
	  Резервные копии только для копирования позволяют делать резервную копию любого типа без каких-либо других резервных копий.
	  
	  Резервные копии только для копирования не влияют на цепочку восстановления.
    - РЕЗЕРВНЫЕ КОПИИ ЖУРНАЛОВ ТРАНЗАКЦИЙ.
	  Резервные копии журналов транзакций записывают записи, 
	  записанные в файл журнала транзакций, определенные для базы данных. 
	  
	  Полные и массовые журналы восстановления - это единственные модели, 
	  поддерживающие резервное копирование журналов транзакций.

ТИПЫ ВОССТАНОВЛЕНИЯ РЕЗЕРВНЫХ КОПИЙ:
    - Восстановление всей базы данных.
    - Частичное восстановление.
    - Восстановление файла или страницы из резервной копии.
    - Восстановление журнала транзакций.
    - Восстановление базы данных до момента времени, используя моментальный снимок базы данных.
	 
МОДЕЛИ ВОССТАНОВЛЕНИЯ:
	- ПОЛНАЯ.
	  База данных, установленная для полного восстановления, 
	  будет иметь все операции с базой данных, записанные в журнал транзакций.
	- ПРОСТАЯ.
      В этом режиме ваш журнал транзакций усекается автоматически на основе контрольной точки в базе данных.
      
      Нет возможности создавать резервные копии журнала транзакций.
	- НЕПОЛНОЕ ПРОТОКОЛИРОВАНИЕ.
	  Модель с неполным протокалированием похожа на полное восстановление, 
	  но она отличается тем, как массовые операции фиксируются в журнале транзакций.
	  
	  Операции массового импорта включают в себя BCP и BULK INSERT. 
	  SELECT INTO, CREATE INDEX И ALTER INDEX могут быть также массовыми операциями при выполнении определенных условий.
	  
	  Модель неполного протаколирования отслеживает только экстенты, 
	  которые были модифицированы операцией массовой загрузки, 
	  но не записывает каждую строку в журнал транзакций.
      
      В данной модели восстановления, резервные копии журналов для баз данных могут быть большими, 
      потому что SQL Server копирует все экстенты данных, 
      на которые влияли операции массовой загрузки с момента последней резервной копии журнала транзакций.
      
      Возможно, что некоторые INSERT INTO ... SELECT операторы будут минимально регистрироваться. 
      Чтобы выполнить эти минимально зарегистрированные вставки, должно быть выполнено следующее:
          1. База данных находится в режиме неполного или простого протоколированием.
          2. Блокировка уровня таблицы приобретается в целевой таблице.
          3. В таблице нет индексов или только пустой кластеризованный индекс.
          4. Таблица не реплицируется.
		  5. В целевой таблице нет триггеров.
          
Единственный метод усечения журнала транзакций - 
это переключение базы данных на использование простой модели восстановления.

РЕЗЕРВНЫЕ КОПИИ ТОЛЬКО ДЛЯ КОПИРОВАНИЯ (см. Резервное копирование\Резервные копии только для копирования.sql).
    Если вы хотите создать резервную копию, которая не повлияет на будущие или прошлые резервные копии,
    существуют резервные копии только для копирования.
    
    Резервная копия только для копирования позволяет сделать резервную копию базы данных или журнала,
    не указывая при этом резервную копию, которая должна быть включена в последовательность восстановления.
