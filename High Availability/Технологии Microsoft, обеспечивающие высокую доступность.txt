Отказоустойчивая кластеризация Windows Server (WSFC).
    Обеспечивает локальную высокую доступность на уровне экземпляра сервера,
    в основном позволяя связать несколько узлов (экземпляров серверов) в активных/активных и активных/пассивных режимах.
    
    Экземпляр отказоустойчивого кластера (FCI) - это единственный экземпляр SQL Server, который устанавливается через узлы WSFC и, возможно, через несколько подсетей.
    
    WSFC позволяет группировать до 64 узлов (на Hyper-V) с некоторыми ограничениями. Обычно группируется между 2 и 4 узлами, обеспечивая как мощность, так и доступность, необходимые для приложения.
    Каждый узел в кластере получает информацию о каждом другом узле таким образом, что если один из них выходит из строя, другой берет на себя его ресурсы (например, его общий диск).
    
    Примером приложения, поддерживающего кластер, является SQL Server.    
    Это считается основополагающим требованием для многих высокодоступных решений, таких как кластеризация SQL и группы доступности AlwaysOn.
SQL кластеризация. 
    Вы можете определить от двух до восьми SQL серверов, запущенных на разных компьютерах, чтобы действовать как точки отказа, если активный SQL сервер когда-либо терпит неудачу.
    
    Как правило, он работает в активном/пассивном режиме (но не ограничивается этим), эта возможность поддержки кластеров гарантирует, посредством создания виртуального SQL Server,
    что клиентское приложение может подключиться и работать на этом виртуальном SQL Server почти все время.
Группы доступности AlwaysOn.
    Функция AlwaysOn основывается на FCS по узлам и обеспечивает мощный синхронный или асинхронный вариант для базы данных и экземпляров, которые могут быть доступны практически без простоя.
    
    Время отказоустойчивости часто измеряется в секундах, а не в минутах или часах. Этот параметр может создавать множество опций для HA и распределенной обработки. 
    С SQL Server 2016 может быть создано восемь вторичных элементов в составе одной группы доступности.
Снимки базы данных.
    Вы можете использовать возможности моментального снимка базы данных в определенный момент времени, чтобы создавать вехи при создании массовых обновлений, вставок или удалений.
    
    Этот метод дает возможность восстановить ваши данные в случае возникновения проблемы без необходимости восстановления базы данных или ее отправки в автономном режиме, тем самым поддерживая доступность вашего приложения.
Репликация данных
    SQL Server предлагает высокоэффективный механизм распространения данных в других местах с целью максимизации доступности и снижения риска сбоев.
    
    Репликация данных определяет модель издателя, распространителя и подписчика для распространения данных (копировать их) в другое место с полной гарантией целостности данных. 
    
    Таким образом, отдельное местоположение может иметь точный образ первичной базы данных, который будет использоваться в качестве точки перехода на другой ресурс или в качестве местоположения, 
    чтобы максимизировать региональную доступность.
Log Shipping
    SQL Server предоставляет механизм, который позволяет использовать журнал транзакций первичной базы данных для вторичной копии той же базы данных. 
    
    Он доступен в качестве точки перехода на другой ресурс, если это необходимо, или в качестве места, чтобы рассмотреть возможность изолировать обработку только для чтения от основного сервера для коэффициентов производительности и доступности.
Другие гибридные или полные опции. 
    Доступны другие параметры, такие как динамическое растяжение теплых и холодных данных в Azure (облако) с помощью Stretch Database и групп доступности, 
    развертывание больших параметров данных (на Azure), полное развертывание баз данных Azure SQL и несколько другие варианты HA в облаке.
