ЗЕРКАЛИРОВАНИЕ БАЗ ДАННЫХ ПОЛНОСТЬЮ ПОДДЕРЖИВАЕТСЯ В SQL Server Standard Edition, Developer Edition и Enterprise Edition, но оно не поддерживается в SQL Server Express Edition. 
Машины с SQL Server Express Edition могут использоваться в качестве серверов-свидетелей.

Когда вы зеркалируете базу данных, вы, запрашиваете полную копию базы данных (нет поддержки для фильтрации, подмножества или любой формы разбиения).

Зеркалирование базы данных работает через журнал транзакций основной базы данных.
Можно зеркалировать только базу данных, использующую полную модель восстановления.

Зеркалирование базы данных использует фактические физические записи журнала как на стороне основного сервера базы данных, 
так и на стороне сервера зеркальной базы данных (записи журнала сжимаются на главной стороне перед передачей).

Типичная конфигурация зеркального отображения базы данных, 
которая состоит из трех компонентов (см. рис. Базовая конфигурация зеркального отображения базы данных.jpg):
    - Основной сервер базы данных. 
      Это источник зеркалирования. 
      Вы можете зеркалировать одну или несколько баз данных на одном экземпляре SQL Server на другой экземпляр SQL Server. 
      Вы не можете зеркалировать базу данных на, тот же экземпляр SQL Server.
    - Зеркальный сервер базы данных.
      Зеркальный сервер является получателем зеркалирования с основного сервера базы данных. 
      
      Эта зеркальная база данных хранится в горячем режиме и не может использоваться напрямую (ее статус находится в режиме непрерывного восстановления).       
      Причина в том, что записи физической транзакции постоянно применяются к этой зеркальной базе данных.
      
      Эта база данных по существу является горячей резервной базой данных и недоступна для прямого использования. 
      Она используется в случае, если главный терпит неудачу, и она не должна быть испорчена каким-либо образом. 
      
      Единственным исключением из этого сценария является - создание моментальных снимков базы данных из зеркала базы данных.
    - Сервер базы данных свидетеля.
      Вы используете сервер базы данных свидетелей, который является необязательным, если вы хотите постоянно проверять, 
      произошли ли какие-либо сбои на главном сервере базы данных и чтобы было принято решение о переходе на сервер зеркальной базы данных. 
      
      С сервером-свидетелем формируется кворум (то есть два из трех серверов), и он принимает кворум для принятия решения об отказе. 
      Типичным сценарием является то, что главный сервер по какой-то причине выходит из строя, свидетель видит эту ошибку, зеркало также видит провал, 
      и вместе они согласны с тем, что он потерян и что зеркало должно взять на себя основную роль. 
      Если свидетель все еще видит, что он жив и здоров, но связь между зеркалом и принципалом была нарушена, 
      свидетель не соглашается переходить к зеркалу (хотя зеркало думает, что оно должно это сделать, потому что оно потеряло связь с основным сервером). 
      Серверы свидетелей обычно помещаются на отдельные физические серверы.
      
С зеркалированием базы данных у вас есть возможность развертывания в одном из трех режимов: 
    - Высокая безопасность с автоматическим режимом перехода на другой ресурс (высокая доступность с сервером-свидетелем).
    - Высокая безопасность без автоматического режима отказа (высокая защита без сервера-свидетеля)
    - Высокая производительность.
      
Зеркалирование базы данных выполняется с использованием асинхронных или синхронных операций:
    - Синхронные операции. 
      При синхронных операциях транзакция фиксируется (то есть, написана) для обоих партнеров пары зеркалирования базы данных.
      Это, очевидно, добавляет некоторую задержку к полной транзакции, поскольку она находится на двух серверах. 
      Режимы высокой безопасности используют синхронные операции.
    - Асинхронные операции.
      С асинхронными операциями транзакции фиксируются, не дожидаясь, пока зеркальный сервер будет записывать журнал на диск.
      Это может значительно повысить производительность. Высокопроизводительный режим использует асинхронные операции.
      
Переключение ролей - это акт передачи главной роли зеркальному серверу. 
Когда происходит сбой, основная роль переключается на зеркальный сервер, 
и его база данных выводится онлайн как основная база данных.

Варианты отказоустойчивости включают следующее:
    - Автоматический переход на другой ресурс.
      Автоматический переход на другой ресурс включен с трехсерверной конфигурацией с участием принципала, зеркала и сервера-свидетеля.
      Требуются синхронные операции, и зеркальная база данных должна быть уже синхронизирована (то есть, синхронно с транзакциями, когда они записываются в принципал).
      Переключение ролей выполняется автоматически.
    - Ручной переход на другой ресурс.
      Необходим ручной переход на другой ресурс, когда нет сервера-свидетеля, и вы используете синхронные операции.
      Переключение ролей выполняется вручную.
    - Принудительное обслуживание.
      Если зеркальный сервер доступен, но, возможно, не синхронизирован, 
      зеркальный сервер может быть принудительно захвачен, когда главный сервер потерпел неудачу. 
      Это может означать потерю данных, поскольку транзакции не были синхронизированы.
